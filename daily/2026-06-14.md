## 遇到的坑与解决方案

| # | 问题 | 原因 | 解决 |
|---|------|------|------|
| 1 | GitHub 下载全部超时 | 网络环境封禁 GitHub 443 端口 | GitHub API 能用，`api.github.com` 走不同 CDN，走 API 拉取信息和下载 |
| 2 | `codex exec` 启动后挂住 | stdin 不是终端 + 等待审批 | 用 `Write-Output "y" \| codex exec ...` 管道输入 |
| 3 | `wire_api = "chat"` 报错 | Codex v0.139.0 移除了 chat wire，只保留 responses | 改用 mimo2codex 做协议翻译代理 |
| 4 | `developer` role 不被 DeepSeek 识别 | Codex Responses API 有 developer 角色，DeepSeek Chat API 不支持 | mimo2codex 内置了角色映射（developer → system） |
| 5 | 自写代理 `codex-deepseek-proxy.mjs` 翻译不完整 | 工具格式、流式响应、WebSocket 等问题太多 | **放弃自写**，改用社区维护的 `mimo2codex` |
| 6 | `Start-Process` 无法启动 `.cmd` 包装脚本 | PowerShell `Start-Process` 需要真实 exe，不能直接跑 npm 全局的 .cmd | 改用 `node.exe` 直接运行 `dist/cli.js` |
| 7 | 桌面快捷方式点击无反应 | CMD 批处理中文编码损坏；powershell 启动 .cmd 方式不对 | 换成 `.ps1` 脚本 + 快捷方式调用 `powershell.exe -File` |
| 8 | Glight 没有 `claude.exe` 进程 | VSCode 扩展版 Claude Code 进程名不是 claude.exe | Hooks 机制不依赖进程检测，状态文件模式正常工作 |

## 经验备忘

1. **这机器上 `api.github.com` 可访问，但 `github.com` 和其 CDN 被墙**——以后拉 GitHub 信息优先走 API
2. **npm 用国内镜像**：`--registry https://registry.npmmirror.com` 比官方源快很多
3. **Codex 版本升级可能带来 breaking changes**——`wire_api` 从 chat → responses 迁移就是例子
4. **自写代理不如用现成的**——mimo2codex 有完善的协议翻译、工具过滤、Admin UI，省大量时间
5. **Glight 首次运行自动配 Hook，不用手动改 settings.json**——设计很贴心
