# 早晨例行程序（Morning Routine）

## 概述

每天早上自动执行 6 步流程：发现昨日 → 产出摘要 → 更新进度 → 提取知识 → 生成待办 → 推送飞书。

**触发方式：**
- 自动：每天 9:03 cron 定时触发（持久化，错过会在下次打开 Claude Code 时追赶）
- 手动：说"跑早晨例行"或 `/morning-routine`

**核心原则：** 只读不删、追加不覆盖、不确定就标注"不确定"、不虚构信息。

## 执行流程

### 第 1 步：发现昨日

```
目标：确定"昨天"是什么日子，找到昨天的活动记录
```

1. 计算昨天日期：`$yesterday = (Get-Date).AddDays(-1).ToString('yyyy-MM-dd')`
2. 检查 `daily/$yesterday.md` 是否存在
3. **如果存在**：读取全文，提取各节内容（完成事项、遇到的坑、关键文件清单、待办/遗留、经验备忘）
4. **如果不存在**：
   - 用 `Get-ChildItem -Recurse -File | Sort-Object LastWriteTime -Descending | Select-Object -First 30 FullName, LastWriteTime` 获取最近修改的文件
   - 排除 `.git/`、`node_modules/`、`.claude/` 下的非配置文件
   - 从文件活动推断昨天做了什么，生成简要记录
5. **如果是周一**（昨天是周日）：额外读取上 `daily/上周五.md`，汇总周末状态变化
6. **如果是月初第一天**：额外回顾上个月进度，在摘要中提及月度小结

输出：对"昨天做了什么"有清晰认知（无论是从日志还是从文件活动推断）。

### 第 2 步：产出昨日摘要

```
目标：为昨天生成/补充一句话摘要，让用户一秒回忆起昨天做了什么
```

1. **如果 `daily/$yesterday.md` 存在且内容完整**：
   - 检查文件顶部是否已有摘要块（`> 摘要：` 开头的引用行）
   - 如果没有，在文件标题下方插入摘要块：
     ```markdown
     > **摘要(YYYY-MM-DD)**: 昨天完成了 [N] 件事：[要点1]、[要点2]、[要点3]。遗留 [N] 项待办。
     ```
   - 摘要控制在 2 行以内，只写最重要的
   - 用 Edit 工具在 `# 每日总结 — YYYY-MM-DD` 标题下方插入

2. **如果 `daily/$yesterday.md` 不存在**：
   - 创建 `daily/$yesterday-summary.md`（轻量版）：
     ```markdown
     # 昨日摘要 — $yesterday
     
     > ⚠️ 昨日无完整日志，以下从文件活动推断。
     
     ## 推断活动
     - [从文件修改时间推断的内容]
     
     ## 文件变更
     | 文件 | 最后修改 |
     |------|----------|
     | ... | ... |
     ```
   - 不要虚构不确定的内容

### 第 3 步：更新进度追踪

```
目标：保持 progress.md 反映项目最新状态
```

1. **读取 `progress.md`**：
   - 如果不存在：使用下方模板创建（参考第 6 节）
   - 如果存在：解析各节内容

2. **标记完成项**：
   - 从昨日日志"完成事项"中提取已完成的任务
   - 在 `progress.md` 的"进行中"找到对应项，移到"已完成"并标注完成日期
   - 如果完成事项不在 progress.md 中（新冒出来的），在"已完成"新增

3. **合并待办**：
   - 昨日日志"待办/遗留"中未完成项 → 合并进 progress.md "进行中"
   - 去重：如果已存在于"进行中"，不重复添加

4. **检测阻塞项**：
   - 遍历"进行中"和"阻塞项"
   - 如果某项超过 7 天没有进展（对比最后更新日期），在备注中标注 `⚠️ 停滞 N 天`
   - 如果某项被标记为完成，但还在"进行中"，移动到"已完成"

5. **更新元信息**：
   - 更新 `> 最后更新: YYYY-MM-DD HH:MM` 时间戳
   - 如果项目阶段发生变化（如：探索期 → 开发期），更新"当前状态"中的阶段描述

6. **更新"最近完成"**：
   - 把最近 3 天内完成的关键事项摘要写在"当前状态 > 最近完成"行

### 第 4 步：提取有用知识到 archive

```
目标：把每日日志中的经验和踩坑按主题归档，形成可检索的知识库
```

1. **解析经验备忘**（从昨日日志"经验备忘"章节）：
   - 每条经验提取为一个独立知识点
   - 为每条经验确定一个主题 slug（英文，kebab-case，如 `github-api-access`、`npm-mirror-config`）
   - 目标路径：`archive/insights/<主题slug>.md`

2. **解析踩坑记录**（从昨日日志"遇到的坑与解决方案"表格）：
   - 表中每一行是一个踩坑记录
   - 为每个记录确定主题 slug（如 `codex-wire-api-migration`、`powershell-start-process`）
   - 目标路径：`archive/pitfalls/<主题slug>.md`

3. **写入策略**（追加模式，不覆盖）：
   - 如果目标文件已存在：追加新的日期段落（去重 — 相同内容不重复写）
   - 如果不存在：创建新文件，包含标题和第一条记录
   - 使用 Write 工具创建新文件，Edit 工具追加到已有文件

4. **归档文件格式**：

   `archive/insights/<topic>.md`：
   ```markdown
   # <主题标题>
   
   ## YYYY-MM-DD
   - 发现: <经验内容>
   - 来源: daily/YYYY-MM-DD.md
   ```

   `archive/pitfalls/<topic>.md`：
   ```markdown
   # <主题标题>
   
   ## YYYY-MM-DD
   - 问题: <问题描述>
   - 原因: <根本原因>
   - 解决: <解决方案>
   - 教训: <可复用的教训>
   - 来源: daily/YYYY-MM-DD.md
   ```

5. **去重规则**：
   - 如果目标文件中已有几乎相同的记录（相似度 > 80%），跳过
   - 如果是同一主题的新发现，追加新日期段落
   - 在文件顶部维护一个 `> 最后更新: YYYY-MM-DD` 时间戳

6. **如果昨日日志无经验备忘/踩坑**：
   - 跳过此步，不强行编造
   - 如果连续 3 天无新知识归档，在 TODO.md 备注中提醒"近期无知识沉淀"

### 第 5 步：生成今日待办

```
目标：生成今日日志模板 + 今日 TODO 清单
```

1. **生成 `daily/YYYY-MM-DD.md` 模板**（今天日期）：
   - 如果已存在（今天已经运行过）：跳过创建，只做增量补充
   - 如果不存在：创建新文件，格式沿用用户 `daily/2026-06-14.md` 的结构：

   ```markdown
   # 每日总结 — YYYY-MM-DD
   
   ## 完成事项
   
   ### 1. 
   - 
   
   ---
   
   ## 遇到的坑与解决方案
   
   | # | 问题 | 原因 | 解决 |
   |---|------|------|------|
   | 1 | | | |
   
   ---
   
   ## 关键文件清单
   
   | 文件 | 用途 |
   |------|------|
   
   ---
   
   ## 待办 / 遗留
   
   - [ ] 从昨日继承：[列出昨日未完成的待办项]
   - [ ] 
   
   ---
   
   ## 经验备忘
   
   1. 
   ```

2. **生成/覆盖 `TODO.md`**（项目根目录）：

   ```markdown
   # 今日待办 — YYYY-MM-DD
   
   > 生成: HH:MM | 继承自 daily/YYYY-MM-DD.md | progress.md
   
   ## 🔴 优先 (今天必须推进)
   - [ ] <从"阻塞项"或昨日遗留中最重要的 1-2 项>
   - [ ] <从 progress.md "进行中"取最紧迫的项>
   
   ## 🟡 其次 (有时间就做)
   - [ ] <其他进行中项，不超过 3 项>
   
   ## 🔵 日常
   - [ ] 更新 daily/YYYY-MM-DD.md（完成事项、遇到的坑、经验备忘）
   
   ## 📝 昨日回顾
   > <昨日一句话摘要>
   ```

3. **TODO 优先级判断规则**：
   - 🔴 优先：阻塞项 + 有明确 deadline 的项 + 昨日遗留的核心项（最多 2-3 项）
   - 🟡 其次：进行中的其他项 + 日常维护（最多 3 项）
   - 🔵 日常：每日必做（写日志），固定不变
   - 总项数控制在 5-7 项，一屏看完

4. **如果是周末**（周六/周日）：
   - 仍然生成 `daily/今天.md` 模板（用户可以填，也可以不填）
   - `TODO.md` 改为"周末自由安排"模式，不强制待办：
     ```markdown
     # 周末 — YYYY-MM-DD
     
     > 🏖️ 周末自由安排。以下是可选事项，不强制。
     
     ## 📝 可选
     - [ ] 更新 daily 日志（如有值得记录的事）
     - [ ] 回顾本周 progress.md，思考下周方向
     ```

### 第 6 步：推送飞书

```
目标：把早晨例行摘要推到飞书，手机上就能看到
```

1. 用 PowerShell 调用 `scripts/send-feishu.ps1`，传入摘要消息
2. **在此之前**，先运行 `scripts/daily-check.ps1` 监测关键网站状态：
   ```powershell
   powershell -NoProfile -ExecutionPolicy Bypass -File "scripts/daily-check.ps1"
   ```
   - 如果某个站点 DOWN，飞书消息中标注 ⚠️
   - 如果某个 SPA 站点检测到关键词命中（非 JS-Rendered），飞书消息中标注 🔔
   - 正常状态无需在飞书中体现，保持消息简洁
3. 消息格式（简洁，手机一屏看完）：

```
☀️ 早晨例行 — 6月18日 09:03

📝 昨日: 完成了早晨自动化系统搭建。遗留夸克浏览器重装。

📊 进度: 已完成 4 项 | 进行中 2 项 | 阻塞 1 项

📋 今日:
  🔴 夸克浏览器重新下载安装
  🔴 早晨自动化系统 Cron 创建和验证
  🟡 PDF 分析工具链整理合并
  🟡 OCR 识别准确率测试
  🟡 套利跟盘 — https://www.jisilu.cn/data/qdii/

🔍 站点监测: 4/4 在线
```

3. **套利提醒**：若当天 TODO 含套利任务，飞书消息中必须附带集思录 QDII 链接 `https://www.jisilu.cn/data/qdii/`，方便手机一键点击。
4. **事招雷达抓取**：每天运行 `node scripts/qgsydw-scraper.js`，从 qgsydw.com 抓取广州各区招聘公告。筛选条件：非教师、非高管、非应届、行政管理/雇员/社区类岗位。有匹配在飞书消息中附带岗位+日期+链接，无匹配显示「今日无匹配」。
5. **DeepSeek 用量分析**：运行 `node scripts/deepseek-usage-scraper.js` 抓取 Token 消耗数据。结合昨日 daily 日志中的完成事项，关联分析：
   - 昨日哪些任务/操作消耗最多（对比日常基线）
   - 是否有异常暴涨（单日消耗超过前7天均值 50%）
   - 优化建议：长对话及时总结、确定性操作用代码、避免重复读取
   - 飞书推送格式：`📊 Token: 昨日 XX万 | 本月 XX万 | 余额 ¥XX | 异常: 无/⚠️XX任务暴涨`
6. **站点监测与招聘搜索**：运行 `powershell scripts/daily-check.ps1` + WebSearch
5. **招聘搜索（用户运营方向）**：每天执行 WebSearch，搜索词：
   ```
   广州 天河 海珠 荔湾 番禺 越秀 用户运营 会员运营 私域运营 社区运营 招聘 2026
   ```
   - 搜索平台自动覆盖：猎聘、51job、智联、俊才网、前程无忧
   - 筛选条件：薪资 ≥10K（底线）、非应届可报、本科起、中小企业优先
   - 排除：大型企业（阿里/腾讯/字节/美团/京东/百度/网易/唯品会/万达/蓝月亮）、国企、上市公司、编制岗
   - 重点看：创业公司/A轮B轮/中小民企/本地企业
   - 原因：中小企业背调概率低
   - 有匹配的在飞书消息中附带岗位 + 薪资 + 链接（最多 3 条）
   - **飞书消息末尾固定附带 Boss直聘直达链接**：
     ```
     💼 Boss: 用户运营 → https://www.zhipin.com/web/geek/job?city=100010000&query=用户运营
     💼 Boss: 会员运营 → https://www.zhipin.com/web/geek/job?city=100010000&query=会员运营
     💼 Boss: 私域运营 → https://www.zhipin.com/web/geek/job?city=100010000&query=私域运营
     ```
   - **飞书消息末尾固定附带常用链接**：
     ```
     🔗 DeepSeek用量 → https://platform.deepseek.com/usage
     🔗 集思录QDII → https://www.jisilu.cn/data/qdii/
     🔗 海珠招聘 → https://hzsg2026.zhaopin.com/zk/
     ```

6. **招聘搜索（事业单位方向）**：每天执行 WebSearch，搜索词：
   ```
   天河区 海珠区 荔湾区 番禺区 越秀区 街道 社区 雇员 管理岗 招聘 本科 2026
   ```
   - 筛选条件：行政管理/公共管理相关、本科起、非应届可报、非教师、非党员可报
   - 有匹配附带岗位名称 + 链接（最多 2 条）

3. 发送命令：
   ```powershell
   powershell -NoProfile -ExecutionPolicy Bypass -File "scripts/send-feishu.ps1" -Message "<消息内容>"
   ```

4. 如果发送失败：不阻塞流程，在终端输出中标注 `⚠️ 飞书推送失败: <原因>`，用户下次打开 Claude Code 仍能看到完整报告。

## 边缘情况速查表

| 场景 | 检测方式 | 处理 |
|------|----------|------|
| 首次运行（无 daily/ 历史） | `daily/` 为空或不存在 | 创建目录，初始化 progress.md 模板，提示用户补充项目概况 |
| 昨日无日志 | `daily/$yesterday.md` 不存在 | 从文件修改时间推断活动，生成 `$yesterday-summary.md` |
| 今天已运行过 | `daily/$today.md` 存在且内容非空模板 | 跳过模板生成，只做增量归档 |
| 周末 | `(Get-Date).DayOfWeek` in `Saturday, Sunday` | 跳过强制作业，生成"周末自由安排"模式 |
| 周一 | 昨天是周日 | 额外读取上周五日志，汇总周末状态 |
| 连续多天未运行 | 昨天日志存在但前天不存在 | 逐天补做摘要（最多回溯 7 天），不堆积 |
| progress.md 被手动编辑 | 检查内容结构 | 保留手动内容，只追加更新，不覆盖已有结构 |
| archive/ 主题文件已存在 | 文件存在检测 | 追加模式，按日期分隔，自动去重 |
| 合并冲突（他人/其他进程同时编辑） | Edit 工具报错 | 先 Read 再重试，不强行覆盖 |

## 初始化模板

### progress.md 模板（首次运行时创建）

```markdown
# 项目进度追踪

> 最后更新: YYYY-MM-DD HH:MM

## 当前状态
- 活跃项目: <项目名称>
- 阶段: <探索期 / 开发期 / 维护期>
- 最近完成: <最近 3 天完成的关键事项>

## 进行中
- [ ] 

## 已完成
- [x] 

## 阻塞项
- [ ] 

## 里程碑
- [ ] 
```

### 首次运行时的额外操作

1. 创建 `archive/insights/` 和 `archive/pitfalls/` 目录（如不存在）
2. 创建 `progress.md`（如不存在）
3. 如果 `daily/` 下已有历史日志，从中提取已有经验补充到 archive（回溯初始化）
4. 在输出末尾提醒用户：
   > 📋 早晨例行初始化完成。请检查 `progress.md` 并根据实际情况补充项目概况。

## 执行后输出

每次执行完成后，在对话中输出简要报告：

```
✅ 早晨例行完成 (2026-06-17 09:03)

📝 昨日摘要: 完成了 CLAUDE.md 行为契约、Codex + DeepSeek 接入、Glight 安装。遗留 Obsidian 安装（网络阻塞）。

📊 进度更新:
  - 已完成: 3 项标记为完成
  - 进行中: 2 项持续推进
  - 阻塞: Obsidian 安装（停滞 3 天）

📚 知识归档:
  - insights/github-api-access.md ← 新增
  - pitfalls/codex-wire-api-migration.md ← 新增

📋 今日待办: TODO.md 已生成 (3 优先 / 2 其次 / 1 日常)

📱 飞书推送: 已发送
```

## 行为约束

1. **只读不删**：对 daily/、progress.md、archive/ 只追加和更新，不删除用户写的内容
2. **不确定就标注**：推断的内容加 `⚠️ 推断，未确认` 标签
3. **不虚构**：不要编造不存在的事项、经验、或文件
4. **简洁**：摘要和 TODO 控制在一屏内，不做长篇大论
5. **如果某步骤无内容可处理**：跳过并说明，不强行填充
