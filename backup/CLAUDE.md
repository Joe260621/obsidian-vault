# CLAUDE.md — Behavior Contract

## 基础规则（核心）

1. **先想再动手**：不确定就问，不瞎猜需求
2. **简洁优先**：用最少代码解决问题，不搞过度设计
3. **外科手术式修改**：只改必须改的，不碰无关内容
4. **目标驱动执行**：先定成功标准，做完必须验证

## Token 成本控制（硬性规则 — 优先级高于便利性）

> **背景**：2026-06-20 单日消耗 ¥32.56，根因是两个超长会话（1073 轮 + 726 轮），缓存读取占 80% 费用。
> DeepSeek 每次对话需重读全部历史缓存，会话越长越贵。以下规则强制执行。

15. **会话分段（硬性）**：同一会话超过 **50 轮**（用户+助手合计），必须主动提议分段。做法：总结当前状态→产出结构化接力文档→建议开新会话。用户拒绝才继续，但不能默许默默涨。
16. **话题切换必分段**：用户切换到新话题/新项目时，先封存旧话题（写总结），再提议开新会话处理新话题。不得在同一会话中串项目。
17. **禁止裸 Continue**：用户说"继续"/"Continue"时，先列出上次停下的具体位置和待做事项，请用户给明确指令。不得用模糊的 Continue 推进——每轮都在为整个历史付费。
18. **头脑风暴分阶段**：长头脑风暴必须拆阶段（探索→聚焦→决策→执行），每阶段结束时做总结，下一阶段开新会话。单阶段不超过 50 轮。
19. **成本自觉**：每次响应超过 3000 output tokens 时，在末尾用 `[本轮估算: input≈X万 output≈Y万]` 简短标注。不需要精确，但要有意识。
20. **长会话预警**：当察觉到当前会话已超过 30 轮时，主动提醒用户「已 30+ 轮，建议分段以控制成本」，但不中断当前任务。

## 补充规则

5. **确定性任务用代码**：格式转换等确定性任务优先用代码，别让模型瞎折腾
6. **控制 Token**：任务长了先总结状态，控制 Token 别浪费
7. **选方案给理由**：遇到冲突选一种方案并说明理由，别混着来
8. **先读再写**：写之前先读现有文件，能复用就不新造轮子
9. **测试要验证业务逻辑**：别只测"有返回"
10. **长任务设检查点**：长任务每步做检查点，说不清状态就停
11. **跟随项目习惯**：跟着项目现有习惯走，别偷偷改风格
12. **如实报告**：失败如实说，不确定就写"不确定"，不隐瞒

## 快捷操作

13. **每日启动**：用户输入"1"时，立即调用 morning-routine skill（等同于 /morning-routine），执行早晨自动化流程
14. **每日备份**：每次新增 skill、文件、项目或重要变更后，运行 `.\scripts\backup.ps1` 生成跨电脑迁移备份到 `.\backup\`
